## Lab Summary

## Key Concepts

| Concept | Meaning |
|---|---|
| Amazon GuardDuty | AWS threat detection service that monitors for malicious or suspicious activity. |
| Detector | The GuardDuty resource that represents GuardDuty being enabled in a region. |
| Finding | A security alert generated by GuardDuty. |
| Sample Finding | A simulated GuardDuty finding used for testing and learning. |
| Severity | A numeric score that helps prioritize findings. |
| Low Severity | Usually 1.0–3.9. Suspicious activity that may not require immediate action. |
| Medium Severity | Usually 4.0–6.9. Suspicious activity that should be investigated soon. |
| High Severity | Usually 7.0–8.9. Activity that may indicate compromise or active attack. |
| Threat Detection | Identifying suspicious activity automatically from logs and behavior patterns. |
| SOC | Security Operations Center, where analysts monitor and investigate alerts. |

### Step 3: Check If GuardDuty Is Already Enabled

**What I did:**

### Step 4: Enable GuardDuty

**What I did:**

- Enabled GuardDuty in `us-east-1`.

**Command used:**

```bash
aws guardduty create-detector --enable --region us-east-1
```

**Expected result:**

```json
{
  "DetectorId": "REDACTED_DETECTOR_ID"
}
```

**Result:**

- GuardDuty detector was created successfully.

**Notes:**

- Creating a detector enables GuardDuty in that region.
- GuardDuty begins monitoring in the background after it is enabled.
- Record the detector ID for later commands.

### Step 5: Generate Sample Findings

**What I did:**

- Generated safe GuardDuty sample findings.

**Command used:**

```bash
aws guardduty create-sample-findings --detector-id <YOUR_DETECTOR_ID> --finding-types "UnauthorizedAccess:IAMUser/MaliciousIPCaller.Custom" "Recon:EC2/PortProbeUnprotectedPort" "CryptoCurrency:EC2/BitcoinTool.B!DNS" --region us-east-1
```

**Expected result:**

```text
No output
```

**Result:**

- Sample findings were generated successfully.

**Sample finding types used:**

| Finding Type | What It Simulates |
|---|---|
| `UnauthorizedAccess:IAMUser/MaliciousIPCaller.Custom` | API calls made from a known malicious IP address |
| `Recon:EC2/PortProbeUnprotectedPort` | Port scanning against an EC2 instance |
| `CryptoCurrency:EC2/BitcoinTool.B!DNS` | EC2 communication with a cryptocurrency mining domain |

**Notes:**

- These are simulated findings.
- They are safe and do not represent real attacks against my account.

### Step 7: Get Finding Details

**What I did:**

- Retrieved details for one GuardDuty finding.

**Command used:**

```bash
aws guardduty get-findings --detector-id <YOUR_DETECTOR_ID> --finding-ids <YOUR_FINDING_ID> --region us-east-1 --query "Findings[0].{Type:Type,Severity:Severity,Title:Title,Description:Description}" --output table
```

**Expected result:**

- A table showing:
  - Type
  - Severity
  - Title
  - Description

**Result:**

- Finding details were displayed successfully.

**Notes:**

- The finding type explains the category of threat.
- The severity number helps determine priority.
- The title and description explain what GuardDuty detected.

---

### Step 8: View All Findings with Severity

**What I did:**

- Retrieved multiple finding details in a table.
- Compared each finding’s type, severity, and title.

**Command used:**

```powershell
aws guardduty get-findings --detector-id <YOUR_DETECTOR_ID> --finding-ids $((aws guardduty list-findings --detector-id <YOUR_DETECTOR_ID> --region us-east-1 --query "FindingIds[]" --output text).Split()) --region us-east-1 --query "Findings[].{Type:Type,Severity:Severity,Title:Title}" --output table
```

**Expected result:**

- A table showing multiple GuardDuty findings and severity values.

**Result:**

- Multiple sample findings were displayed with severity levels.

**Notes:**

- This command is written for PowerShell.
- The `.Split()` part converts the text list of finding IDs into separate values for the `--finding-ids` parameter.

### Step 10: Understand Severity Levels

**What I reviewed:**

| Severity Range | Level | Meaning | Response Priority |
|---|---|---|---|
| 7.0–8.9 | High | Resource may be compromised or under active attack | Investigate immediately |
| 4.0–6.9 | Medium | Suspicious behavior or deviation from normal activity | Investigate soon |
| 1.0–3.9 | Low | Suspicious but lower-risk activity | Review when possible |

**Real-world interpretation:**

| Level | Example | Meaning |
|---|---|---|
| High | `CryptoCurrency:EC2/BitcoinTool.B!DNS` | An EC2 instance may be compromised and communicating with a mining domain |
| Medium | `UnauthorizedAccess:IAMUser/MaliciousIPCaller` | Credentials may be used from a suspicious IP |
| Low | `Recon:EC2/PortProbeUnprotectedPort` | External reconnaissance or scanning activity may be occurring |

**Notes:**

- High findings usually require immediate incident response.
- Medium findings should be investigated soon.
- Low findings may be reviewed during routine security monitoring.

## Troubleshooting Notes

| Issue | What It Means | How to Fix |
|---|---|---|
| GuardDuty detector already exists | GuardDuty is already enabled in `us-east-1` | Run `aws guardduty list-detectors --region us-east-1`, record the detector ID, and continue to sample findings |
| `create-sample-findings` fails | Detector ID may be wrong or missing | Confirm the detector ID from `list-detectors` or `create-detector` |
| `list-findings` returns an empty list | Sample findings may not be ready yet | Wait 30–60 seconds and run the command again |
| `get-findings` fails | Finding ID may be copied incorrectly | Re-run `list-findings` and copy the full finding ID |
| Console shows no findings | Console may be in the wrong AWS region | Switch the console region to `us-east-1` |
| SSO token expired | AWS CLI SSO session expired | Run `aws sso login --profile <YOUR_PROFILE_NAME>` |
| Cleanup fails | Wrong detector ID or wrong region | Confirm detector ID and use `--region us-east-1` |

- Sample findings are safe simulations and do not mean the account was attacked.
- Severity levels help security teams prioritize alerts.
- High findings usually require immediate response.
- Medium findings should be investigated soon.
- Low findings should still be reviewed but may be lower priority.
- The AWS Console gives a SOC-style view of findings.
