# Lab 06: Serverless File Pipeline

## Lab Summary

In this lab, I built a serverless file processing application with a web frontend. The application allows a user to submit text through a static website, upload the text to an S3 input bucket using a presigned URL, trigger a Lambda function when the file is uploaded, process the file, and save the processed result to an S3 output bucket.

This lab is important because it demonstrates a real serverless architecture pattern where files are uploaded, processed, and returned without managing any servers.

## Architecture Overview

```text
User Browser
    |
    | opens static website
    v
S3 Website Bucket
    |
    | requests upload URL
    v
Presign Lambda Function URL
    |
    | returns temporary upload URL
    v
S3 Input Bucket
    |
    | object created event
    v
Processor Lambda Function
    |
    | writes processed file
    v
S3 Output Bucket
    |
    | browser fetches result
    v
User Browser
```

## Key Concepts

| Concept | Meaning |
|---|---|
| Serverless | A cloud model where code runs without managing servers. |
| Lambda Function | A small unit of code that runs when triggered. |
| Presigned URL | A temporary URL that allows a specific S3 action, such as uploading one file. |
| Lambda Function URL | A public HTTP endpoint for directly invoking a Lambda function. |
| S3 Event Notification | A bucket event that can trigger Lambda when an object is created. |
| Input bucket | S3 bucket where uploaded files first land. |
| Output bucket | S3 bucket where processed files are saved. |
| Website bucket | S3 bucket that hosts the static frontend. |
| CORS | Browser security control that allows or blocks cross-origin requests. |
| Environment variable | A configuration value passed into Lambda, such as a bucket name. |
| CloudWatch Logs | Logs generated by Lambda executions for troubleshooting. |

## Security Notes

| Topic | Explanation |
|---|---|
| Presigned URLs | The browser does not receive AWS access keys. It only receives a temporary upload URL. |
| Temporary access | The upload URL expires after a short period, limiting exposure. |
| Public Function URL | The presign function is public for the lab. In production, authentication should be considered. |
| Public output bucket | The output bucket is publicly readable for the lab so the browser can display results. Do not store sensitive data there. |
| S3 full access policy | The lab uses broad S3 permissions for simplicity. Production should use least privilege. |
| CORS wildcard | The lab allows broad CORS access. Production should restrict allowed origins. |
| Cleanup | Public buckets and public Lambda URLs should not be left active longer than needed. |

## Issues Encountered

| Issue | Cause | Fix |
|---|---|---|
| Lambda Function URL returned `403 Forbidden` | The Function URL had `AuthType: NONE`, but the Lambda resource-based policy did not include all required public invoke permissions | Added an additional permission for `lambda:InvokeFunction` using `--invoked-via-function-url` |
| Upload failed with `405 Method Not Allowed` | The upload flow broke because the browser could not get a valid presigned URL from Lambda. The browser then attempted to send a `PUT` request to the S3 static website endpoint, which does not allow uploads | Fixed the Lambda Function URL permission issue, then confirmed the upload used the presigned S3 URL instead of the website endpoint |


## Troubleshooting Notes

| Issue | Possible Cause | Fix |
|---|---|---|
| Website loads but upload fails | Lambda Function URL, CORS, or presigned URL issue | Check browser console, Function URL, and Lambda CORS settings |
| `AccessDenied` when uploading to S3 | CORS, bucket permissions, or presigned URL issue | Confirm input bucket CORS and presign Lambda environment variable |
| Processor Lambda does not run | S3 notification or Lambda invoke permission missing | Confirm `s3-notification.json` and `lambda add-permission` were applied |
| Output file is missing | Processor Lambda failed or output bucket variable is wrong | Check CloudWatch Logs and `OUTPUT_BUCKET` environment variable |
| Website cannot display processed file | Output bucket is not publicly readable or CORS is missing | Confirm output bucket policy and CORS |
| Lambda function creation fails | IAM role ARN, zip file, or handler name is wrong | Verify role ARN, file names, zip files, and handler values |
| `NoSuchBucket` | Bucket name was typed incorrectly | Check bucket names with `aws s3 ls` |
| Browser CORS error | Required service does not allow cross-origin browser calls | Check CORS on Lambda Function URL, input bucket, and output bucket |
| `403 Forbidden` from Lambda Function URL | The request was blocked before the Lambda code ran | Tested the Function URL directly in PowerShell with `Invoke-WebRequest` |
| `405 Method Not Allowed` from S3 website endpoint | The browser was sending `PUT` to the wrong endpoint | Checked DevTools Network tab and saw the request going to `s3-website-us-east-1.amazonaws.com` |
| Upload request does not go to input bucket | The frontend did not receive or use a valid presigned URL | Checked whether the Lambda response included `uploadUrl` |

## What I Learned

- Serverless applications can process files without managing EC2 instances.
- S3 can host a static website and store application input/output files.
- Lambda can generate presigned URLs for secure temporary uploads.
- Lambda Function URLs can expose a Lambda function through HTTPS.
- S3 event notifications can automatically trigger Lambda when files are uploaded.
- CORS is required when a browser interacts with services on different domains.
- CloudWatch Logs are important for troubleshooting Lambda functions.
- IAM roles allow Lambda to access S3 securely.
- Public buckets and public function URLs should be carefully controlled.
- Cleanup is especially important when a lab creates multiple public resources.
